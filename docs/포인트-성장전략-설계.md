# 뽑뽑 — 포인트·성장전략 설계 문서

> 작성 목적: 배포(상업화)를 향해, "유저를 모으는 유인책 = 포인트"라는 방향을
> 실제로 구현 가능하고 법적으로 안전하게 정리한다.
> 이 문서는 개발 착수 전 **의사결정 문서**다. 코드는 이 문서의 결론을 따른다.

---

## 0. TL;DR (결론부터)

1. **포인트를 "유저↔유저 물건값 결제"에 쓰지 마라.** 그 순간 전자금융거래법상
   *선불전자지급수단*이 될 위험이 크고, 자본금 20억·별도관리 의무가 붙는다.
   → 대신 **에스크로 수수료 할인 / 프리미엄 노출 / 뱃지** 등 "우리가 파는 것"에만 쓰게 한다.
2. **중복 제보는 "막지 말고 병합하라."** 거리+이름 유사도로 후보를 띄우고,
   유저가 "같은 곳이에요"를 누르면 병합. 최초 제보자에게 포인트를 몰아주고,
   중복은 "보강(confirm)"으로 처리해 소액 포인트를 준다.
3. **신선도는 "블러"가 아니라 "확신도(confidence) 배지 + 재인증 리워드"로.**
   블러는 UX를 해치고 이탈을 부른다. 대신 오래된 정보는 흐릿한 배지("○일 전 확인")를
   달고, 누구든 "지금도 있어요/바뀌었어요"를 누르면 포인트를 준다.
4. **콜드스타트는 "공개 데이터 시딩 + 시드 유저 + 얼리버드 보상"의 3중 조합.**
   공개 데이터는 라이선스 확인이 필수. 가장 안전한 건 "가게 자체 정보"(위치·상호)만
   씨딩하고, "지금 뭐가 있는지"는 유저가 채우게 하는 것.

## 1. 포인트 경제 설계

### 1.1 왜 포인트인가 (그리고 왜 그것만으로는 부족한가)

포인트는 훌륭한 **초기 유인책**이지만, 포인트만으로 커뮤니티가 굴러가지 않는다.
포인트는 "행동을 유도"하지만 "머무를 이유"는 만들지 못한다. 아래 두 축을 같이 설계한다.

- **외재적 동기 (포인트)** — 제보/등록/재인증 시 즉시 보상. 초기 행동 점화용.
- **내재적 동기 (지위·소속감)** — 랭킹, 지역 1등 뱃지, "이 동네 뽑기 지도는 내가 만들었다"는
  기여감. 장기 리텐션의 진짜 엔진.

> 핵심 원칙: **포인트는 마중물이고, 커뮤니티 지위가 본체다.**

### 1.2 포인트 적립 정책 (인플레이션 방지 포함)

| 행동 | 포인트 | 제한 / 안전장치 |
|---|---|---|
| **최초** 가게 등록(지도에 없던 곳) | 100 | 중복 판별 통과 시에만. 위치+상호 검증 |
| 제보(기계 라인업/신상) | 30 | 기존 가게에 대한 신규 정보 |
| 중복 제보 → "보강"으로 전환 | 5 | 같은 곳 확인만 해줘도 소액 지급 |
| 신선도 재인증("지금도 있어요") | 10 | 가게당 24h 쿨다운. 남의 정보만 |
| 피드(자랑) 작성 | 5 | **하루 3건까지만**(=15p/일 상한) |
| 좋아요/댓글 받기 | 1 | 하루 상한 20p. 자기 콘텐츠 제외 |
| 신고가 실제 조치로 이어짐 | 20 | 어뷰징 신고엔 0 |

**인플레이션 방지가 경제 설계의 절반이다:**

- **획득 상한(daily cap):** 유저당 하루 총 획득 포인트에 상한(예: 200p).
- **소각(sink)이 반드시 있어야 한다:** 포인트를 버는 곳만 있고 쓰는 곳이 없으면
  가치가 0으로 수렴한다. 아래 1.4의 사용처가 소각 역할.
- **어뷰징 차단:** 자기 제보에 자기가 재인증 금지, 같은 기기/IP 다계정 탐지,
  신규 계정은 첫 24h 적립 제한(가입 즉시 대량 어뷰징 방지).

### 1.2.1 사용처(SINK) 구현 우선순위 — 확정

초기 배포 기준, "구현 난이도 낮음 + 소각 효과 큼 + 법적 안전" 순으로 확정:

1. **[1순위] 마켓 끌어올리기** — 기존 `MarketTab`에 필드 추가만으로 구현. 유저 수요 최강.
   순수 내부용으로 만들되 **나중에 현금 결제를 얹기 쉬운 구조**로 설계
   (차감 로직과 결제수단을 분리 → `spend_points()`가 포인트만 차감, 결제수단은 교체 가능).
2. **[2순위] 뱃지·타이틀 꾸미기** — 원가 0, 리텐션 강함. 지역 랭킹/타이틀과 연동.
3. **[3순위] 포인트 인형뽑기 가챠** — 앱 컨셉과 완벽 일치하나 확률·아이템 설계 필요.
   **Phase 2로 미룸**(현금 환급 없어야 사행성 규제 회피).

### 1.3 포인트 사용처 — 법적 안전지대 안에서

⚠️ **여기가 이 프로젝트에서 법적으로 가장 중요한 갈림길이다.** (자세한 근거는 §4)

전자금융거래법상 포인트가 **선불전자지급수단**이 되는 조건은 아래 **두 가지를 모두** 충족할 때다:
1. 발행인(=뽑뽑) *외의 제3자*로부터 재화·용역을 구입하는 데 사용
2. 구입 가능한 재화·용역이 **2개 업종 이상**

→ 따라서 **"뽑뽑이 직접 제공하는 것"에만 쓰게 하면 규제 밖**이다. 안전한 사용처:

| 사용처 (SINK) | 성격 | 안전한가? |
|---|---|---|
| 프리미엄 노출(내 마켓글 상단 고정) | 뽑뽑이 파는 서비스 | ✅ 안전 |
| 강조 뱃지 / 프로필 꾸미기 | 뽑뽑이 파는 서비스 | ✅ 안전 |
| 광고 제거 | 뽑뽑이 파는 서비스 | ✅ 안전 |
| "이 동네 스폰서" 같은 명예 타이틀 | 뽑뽑이 파는 서비스 | ✅ 안전 |
| 응모권(추첨 이벤트 참여) | 사행성 주의하되 무상경품이면 OK | ⚠️ 조건부 |
| **유저↔유저 마켓 물건값 결제** | 제3자 재화 구입 | ❌ **위험** |
| 현금/기프티콘 환급 | 환금성 | ❌ **위험** |

### 1.4 "마켓 거래 결제"를 원한다면 — 우회 설계

성훈님은 "마켓에서 포인트로 거래"를 원하셨다. 직접 결제는 위험하지만,
**같은 사용자 경험을 규제 밖에서 내는 방법**이 있다:

- **방법 A (권장): 수수료 할인 크레딧.**
  거래는 유저끼리 현금(계좌이체·직거래)으로 하고, 뽑뽑은 "안전거래 수수료"나
  "끌어올리기 비용"을 받는다. 포인트로는 그 **수수료만** 깎아준다.
  → 포인트가 제3자 물건값에 직접 안 쓰이므로 안전.
- **방법 B: 판매자 인센티브를 뽑뽑이 부담.**
  "포인트로 100원 깎기"를 눌러도, 그 100원은 구매자→판매자가 아니라
  뽑뽑이 판매자에게 지급하는 프로모션 크레딧으로 처리. (마케팅비로 회계)
  단 규모가 커지면 비용 부담 → 초기 한정 이벤트로만.
- **방법 C: 규모가 커지면 정식 등록.**
  발행잔액 30억 미만 & 연 총발행 500억 미만이면 **등록 의무 자체가 면제**된다(§4).
  즉 초기엔 방법 A로 가다가, 트래픽이 커지고 진짜 페이 기능이 필요해지면
  그때 전자금융업 등록을 검토. 처음부터 짊어질 필요 없음.

> 요약: **1단계 배포에서는 "포인트=마켓 수수료 할인 + 노출/뱃지"로 시작.**
> 유저↔유저 직접 포인트 결제는 로드맵 뒤로 미룬다.

## 2. 중복 제보 판별 — "막지 말고 병합하라"

성훈님 요구: *"수익도 내야 하지만 유저 편리성도 있어야 한다."*
→ 완전 차단은 편리성을 해치고(진짜 다른 가게인데 막힘), 방치는 데이터 품질을 해친다.
**정답은 "실시간 감지 + 유저가 판단"의 병합 모델.**

### 2.1 등록 순간의 흐름 (UX)

```
유저가 가게 등록 위치를 찍음
        │
        ▼
반경 50m 안 + 상호 유사도 검사
        │
   ┌────┴─────┐
후보 없음     후보 있음
   │            │
   ▼            ▼
신규 등록    "혹시 이 가게 아닌가요?" 카드 노출
(+100p)       ├─ [맞아요, 여기예요] → 기존 가게에 "보강" (+5p)
              └─ [아니요, 다른 곳이에요] → 신규 등록 (+100p)
                    ※단, 남용 방지: 반경 20m 내 동일상호는
                      관리자 검토 큐로 (자동승인 보류)
```

- **최초 등록자만 100p**, 이후 같은 곳 확인은 5p. → 최초 제보 유인 극대화 + 중복 유입도 소액 보상.
- 유저에게 "중복이에요"라고 벌주는 느낌이 아니라 "덕분에 확인됐어요"라고 보상하는 프레이밍.

### 2.2 중복 감지 로직 (의사코드)

```sql
-- 후보 탐색: 지리적으로 가깝고(≈50m) 상호가 비슷한 가게
-- 위경도 1도 ≈ 111km. 위도 0.00045도 ≈ 50m (대략)
SELECT id, place_name, latitude, longitude,
       -- 대략적 거리(m). 정밀도 필요하면 PostGIS earth_distance 사용 권장
       ( 6371000 * acos(
           cos(radians(:lat)) * cos(radians(latitude)) *
           cos(radians(longitude) - radians(:lng)) +
           sin(radians(:lat)) * sin(radians(latitude))
       ) ) AS distance_m
FROM places
WHERE latitude BETWEEN :lat - 0.0006 AND :lat + 0.0006   -- 1차 바운딩박스로 빠르게 필터
  AND longitude BETWEEN :lng - 0.0006 AND :lng + 0.0006
HAVING distance_m < 50
ORDER BY distance_m ASC;
```

```
// 상호 유사도(앱단): 정규화 후 비교
normalize(name):  소문자 + 공백/특수문자 제거 + "점","호점" 등 접미사 정리
score = 1 - (levenshtein(normalize(a), normalize(b)) / max(len))
if distance_m < 50 and score > 0.6:  → 중복 후보로 제시
```

- **1차 필터는 바운딩박스**(인덱스 잘 탐, 빠름), 2차만 정밀 거리·유사도.
- 정밀도가 더 필요하면 Supabase의 PostGIS(`earth_distance`, `ll_to_earth`) 확장을 쓴다.
- 카카오 장소 검색으로 이미 place_id가 잡히면 **place_id 일치 = 확실한 중복**(제일 강력).

### 2.3 스팸/장난 등록 방지

- 반경 20m 내 동일 상호 재등록 → 자동승인 보류, 관리자 큐.
- 위치와 상호가 카카오맵 검색 결과와 매칭 안 되면 "확인 필요" 배지.
- 초기엔 §1.2의 어뷰징 장치(신규계정 제한, 다계정 탐지)로 충분.

## 3. 정보 신선도 — "블러 대신 확신도 배지 + 재인증 리워드"

성훈님 아이디어("시간 지나면 블러, 재인증하면 해제")는 **문제 인식은 정확**하다.
다만 블러는 두 가지 부작용이 있다:
- 유저가 정보를 못 보면 **앱을 열 이유가 사라진다** → 이탈.
- "재인증 안 되면 아예 안 보임"은 콜드스타트 초기에 치명적(대부분이 블러 처리됨).

### 3.1 대안: Confidence(확신도) 상태 머신

정보를 숨기지 않고, **얼마나 믿을 만한지를 시각적으로 표시**한다.

```
[FRESH]  확인 7일 이내      → 초록 "✓ 최근 확인" 배지
   │  (7일 경과)
   ▼
[AGING]  8~30일            → 회색 "○일 전 확인" 배지
   │  (30일 경과)
   ▼
[STALE]  30일+             → 흐린 카드 + "정보가 오래됐어요" + [지금 확인하기] 버튼
   │
   ├─ 누군가 "지금도 있어요" → FRESH로 리셋 (+10p)
   └─ 누군가 "바뀌었어요/없어졌어요" → 신고성 갱신, 라인업 수정 유도(+10p)
```

- **STALE도 정보는 보여준다.** 다만 흐릿하게 + 재인증 CTA를 크게. (블러의 "숨김"이 아니라 "낮은 채도")
- 재인증은 **크라우드소싱 리워드의 핵심 루프**. 이게 §1의 포인트 sink/source를 연결한다.
- "기계 안 상품이 바뀐다" 문제는 특히 **라인업(뭐가 들어있나)** 필드에 적용.
  가게 존재 자체(위치)는 잘 안 변하지만 상품은 자주 변하므로, 신선도 타이머를
  **가게가 아니라 "라인업 제보" 단위로** 건다.

### 3.2 구현 포인트

- `posts`/`reports`에 `last_verified_at`, `verify_count` 컬럼 추가.
- 조회 시 `now() - last_verified_at`으로 상태 계산(서버 or 클라이언트).
- 재인증 버튼 → `last_verified_at = now()`, `verify_count += 1`, 포인트 지급 + 쿨다운 체크.

## 4. 법률 검토 (전자금융거래법 중심)

> ⚠️ 본 절은 공개 자료 기반 정리이며 법률 자문이 아니다.
> 실제 상업화 직전에는 반드시 변호사·법무 검토를 받아야 한다.

### 4.1 핵심 판단 기준

금융위 유권해석상 포인트가 **선불전자지급수단**에 해당하려면 **아래 둘을 모두** 충족해야 한다:

1. **발행인(뽑뽑 및 특수관계인) 외의 제3자**로부터 재화·용역을 구입하는 데 사용
2. 구입 가능한 재화·용역이 **한국표준산업분류 중분류 기준 2개 업종 이상**

→ **뽑뽑이 직접 제공하는 서비스(노출·뱃지·수수료 할인)에만 쓰면 조건 1을 안 건드린다.**
→ 즉 §1.3의 "안전한 사용처"만 열면 규제 대상이 아니다.

### 4.2 소액 예외 (2024.9.15 시행)

설령 선불전자지급수단에 해당하더라도, **발행잔액 30억 원 미만 & 연간 총발행액 500억 원 미만**이면
**등록 의무가 면제**된다. 초기 스타트업은 이 예외 안에 있을 가능성이 높다.
→ 그래도 "구조적으로 규제 밖"인 설계(§1.3)가 훨씬 안전하다. 예외는 규모가 커지면 깨진다.

### 4.3 등록이 필요해질 경우의 요건 (참고)

- 자본금 20억 원 이상, 부채비율 200% 이하
- 전산 인력 5명 이상, 정보보호 체계 등 인적·물적 요건
- 선불충전금 100% 별도관리(신탁·지급보증) 의무

### 4.4 결론 (법률)

- **1단계(배포~초기): "뽑뽑 자체 제공 리워드"에만 포인트 사용.** 규제 밖. 등록 불필요.
- **현금·기프티콘 환급 금지.** 환금성이 생기면 규제·세무 리스크 급증.
- **유저↔유저 직접 포인트 결제 보류.** 하려면 §1.4의 우회(수수료 할인) 또는 정식 등록.
- 배포 시 **약관·개인정보처리방침에 포인트 정책(소멸·유효기간·환불 불가) 명시** 필요.
  (이미 `app/terms`, `app/privacy` 존재 → 포인트 조항 추가 필요)

## 5. 콜드스타트 — 빈 지도를 어떻게 채우나

빈 앱은 첫인상에서 죽는다. "이미 우리 동네가 채워져 있네"가 첫 리텐션을 만든다.
**3중 조합**을 권장한다.

### 5.1 공개 데이터 시딩 — 가능한가? (조건부 가능)

- **가장 안전한 소스: 지자체/공공데이터포털의 인허가·업소 데이터.**
  "게임제공업(청소년게임제공업)", "인형뽑기방" 관련 업소 인허가 데이터가 공공데이터로
  공개돼 있는 경우가 있다. 공공데이터는 대개 이용약관상 재사용이 허용된다(출처 표기).
- **카카오/네이버 지도 크롤링은 위험.** 각사 이용약관상 크롤링·재배포 금지 조항이 있어
  라이선스 위반 소지. 카카오맵은 **공식 로컬 검색 API 범위 내 사용**만 안전
  (성훈님 앱이 이미 쓰는 방식). API 결과를 대량으로 긁어 자체 DB로 저장하는 건
  약관 확인 필수 → 원칙적으로 지양.
- **권장: "가게 존재 정보(상호·위치·주소)"만 공공데이터로 씨딩하고,
  "지금 뭐가 있는지(라인업)"는 비워둔다.** 그래야:
  - 지도가 비어 보이지 않고(가게 핀은 있음),
  - 유저에게 "여기 라인업 첫 제보 = 100p!"라는 명확한 채울 거리를 준다.
  - 라이선스 리스크도 최소(공공데이터 출처 표기).

> ⚠️ 어떤 데이터든 **출처의 라이선스를 반드시 확인**하고 출처를 표기할 것.
> "공공데이터라서 무조건 OK"는 아니다 — 데이터셋별 이용조건이 다르다.

### 5.2 시드 유저 + 직접 시딩

- 성훈님이 아는 지역 5~10곳을 직접 등록해 "밀도 있는 시범 동네" 하나를 만든다.
  전국이 텅 빈 것보다 **한 동네가 꽉 찬 게** 훨씬 강력한 데모다.
- 뽑기 커뮤니티(디시·아카·인스타 해시태그 #인형뽑기)에 시드 유저 모집.

### 5.3 얼리버드 보상

- 출시 초기 N일간 **최초 제보 포인트 2배**, "창립 멤버" 영구 뱃지.
- "이 동네 1호 제보자" 명예 타이틀 → 지역 경쟁 심리 자극(내재적 동기).

## 6. "포인트 말고 다른 유인책" — 추가 아이디어

포인트는 마중물일 뿐, 아래를 겹쳐야 리텐션이 산다:

1. **지위·랭킹:** 지역별 기여 랭킹, "○○동 뽑기 지도 마스터" 타이틀. 사람은 순위에 반응한다.
2. **수집·컬렉션:** 자기가 뽑은 인형을 "도감"처럼 모으는 피드. 게이미피케이션.
3. **실용 정보의 독점성:** "여기 신상 들어왔대"를 제일 먼저 아는 곳 → 정보 자체가 유인.
   알림("찜한 가게에 신상 입고")이 강력.
4. **소셜 그래프:** 채팅·팔로우로 "뽑기 친구"가 생기면 앱을 떠나기 어려워진다(전환비용).
5. **오프라인 연계:** 제휴 가게 "뽑뽑 유저 1회 추가 서비스" 같은 리얼 혜택
   (이건 제휴 협상 필요, 후순위).

> 유인책 우선순위(초기): **얼리버드 포인트 → 지역 랭킹/뱃지 → 신상 입고 알림.**

## 7. 개발 로드맵 (성훈님 순서 반영)

성훈님 계획: *기능 개발 → 고도화 → 보안 점검 → 버그 사냥 → 배포*

### Phase 1 — 포인트·크라우드소싱 코어
- [ ] `points_ledger` 원장 테이블(적립/차감 이력) + `profiles.point_balance`
- [ ] `places` 테이블 분리(가게 = 위치, 제보 = 라인업/신선도)
- [ ] 중복 감지(바운딩박스+유사도) + "이 가게 아닌가요?" UI
- [ ] 신선도 상태(FRESH/AGING/STALE) + 재인증 버튼 + 리워드
- [ ] 적립 상한·쿨다운·어뷰징 방지 서버 로직(RLS + RPC 함수로 서버권한 처리)

### Phase 2 — 사용처(SINK) & 유인책
- [ ] 포인트 사용처: 마켓 끌어올리기 / 뱃지 / 광고제거
- [ ] 지역 랭킹, 얼리버드 뱃지, 신상 입고 알림

### Phase 3 — 콜드스타트
- [ ] 공공데이터 라이선스 확인 → 가게 위치 시딩
- [ ] 시범 동네 직접 시딩 + 시드 유저 모집

### Phase 4 — 고도화 → 보안 → 버그 → 배포
- [ ] 약관/개인정보처리방침에 포인트 조항 추가
- [ ] 보안 점검(§ 아래 별도) → 버그 사냥 → 배포

> **배포 전 필수 보안 이슈(1차 분석에서 발견):**
> - `send-code/route.ts`의 `NODE_TLS_REJECT_UNAUTHORIZED='0'` 제거 (TLS 검증 무력화)
> - 포인트 적립은 **반드시 서버(Supabase RPC/Edge Function)에서** 처리.
>   클라이언트에서 `point_balance`를 직접 UPDATE하게 두면 무한 어뷰징.

## 8. DB 스키마 초안 (Phase 1)

```sql
-- 프로필(포인트 잔액)
create table if not exists profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  nickname text,
  point_balance int not null default 0,
  created_at timestamptz default now()
);

-- 포인트 원장: 모든 적립/차감을 '기록'으로 남긴다(감사·복구·어뷰징 추적)
create table if not exists points_ledger (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users(id) on delete cascade,
  amount int not null,                 -- +적립 / -차감
  reason text not null,                -- 'place_create' | 'report' | 'reverify' | 'feed' | 'spend_boost' ...
  ref_type text,                       -- 'place' | 'post' | 'market' ...
  ref_id text,
  created_at timestamptz default now()
);
create index if not exists idx_ledger_user_time on points_ledger(user_id, created_at desc);

-- 가게(위치 단위) — 제보(라인업)와 분리
create table if not exists places (
  id uuid primary key default gen_random_uuid(),
  place_name text not null,
  address text,
  latitude double precision not null,
  longitude double precision not null,
  kakao_place_id text,                 -- 있으면 중복 판별의 최강 키
  created_by uuid references auth.users(id) on delete set null,
  created_at timestamptz default now()
);
-- 지리 검색 인덱스(바운딩박스용). 정밀도 필요하면 PostGIS 도입.
create index if not exists idx_places_lat on places(latitude);
create index if not exists idx_places_lng on places(longitude);

-- 신선도: 제보(라인업)에 검증 타이머
alter table posts add column if not exists last_verified_at timestamptz default now();
alter table posts add column if not exists verify_count int default 0;
alter table posts add column if not exists place_id uuid references places(id);

-- 포인트 적립은 서버 함수로만(클라이언트 직접 UPDATE 금지)
-- SECURITY DEFINER 함수 + 상한/쿨다운 검증 로직을 여기에.
create or replace function award_points(p_user uuid, p_amount int, p_reason text, p_ref_type text, p_ref_id text)
returns void as $$
begin
  -- TODO: 일일 상한/쿨다운/자기참조 방지 체크
  insert into points_ledger(user_id, amount, reason, ref_type, ref_id)
    values (p_user, p_amount, p_reason, p_ref_type, p_ref_id);
  update profiles set point_balance = point_balance + p_amount where id = p_user;
end;
$$ language plpgsql security definer;
```

> **원장(ledger) 패턴을 쓰는 이유:** 잔액만 저장하면 어뷰징 추적·롤백·정산이 불가능하다.
> 모든 포인트 변동을 append-only 기록으로 남기고, 잔액은 그 합계(또는 캐시)로 본다.
> 회계·분쟁·환불 대응에 필수.
